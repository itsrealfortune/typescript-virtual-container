# Example 25 — VFS Hooks and Resolvers

Registers content resolvers (files dynamically generated on read)
and beforeRead/beforeWrite hooks for monitoring, audit logging, and
access control on the `VirtualFileSystem`.

**Modules:** `VirtualFileSystem`

## The Scenario

The Linux `/proc` filesystem contains dynamically generated files —
`/proc/cpuinfo`, `/proc/meminfo`, `/proc/uptime` — that don't exist as
real files on disk. Reading them triggers a kernel callback that populates
the content on the fly.

Additionally, system administrators often want:

- **Audit hooks** that log every read access to `/etc` (monitoring for
  reconnaissance)
- **Read-only guards** that prevent writes to `/sys` (the kernel's
  virtual filesystem for hardware configuration)
- **Status endpoints** that report real-time application state without
  writing a file to disk (e.g., `/var/www/status.json`)

The `VirtualFileSystem` implements all three patterns through two
mechanisms:

1. **Content resolvers** — functions registered on a path prefix that
   dynamically generate content when a file is read
2. **Lifecycle hooks** — callbacks fired before reads (`onBeforeRead`)
   and before writes (`onBeforeWrite`) on a path prefix

## Modules Used

```typescript
import { VirtualFileSystem } from "../src";

const vfs = new VirtualFileSystem();

vfs.mkdir("/proc");
vfs.mkdir("/sys");
vfs.mkdir("/etc");
```

We create a bare VFS and create three directories that typical Linux systems
have. Note that we don't create files yet — some of them will be dynamically
generated by resolvers.

## Step-by-Step Walkthrough

### Content resolvers

```typescript
vfs.registerContentResolver("/proc", (path) => {
  if (path === "/proc/cpuinfo") {
    return "processor\t: 0\nvendor_id\t: Virtual\nmodel name\t: Virtual CPU\ncpu MHz\t\t: 2400.00\n";
  }
  if (path === "/proc/meminfo") {
    return `MemTotal:       ${512 * 1024} kB\nMemFree:        ${256 * 1024} kB\n`;
  }
  return null; // not handled, fall through to normal VFS
});
```

`registerContentResolver(prefix, callback)` associates a resolver function
with a path prefix (`/proc`). When any file under that prefix is read, the
resolver is called with the full path. The resolver must:

- **Return a string** to provide the file's content (the read succeeds)
- **Return `null`** to indicate "not handled" (the read falls through to
  the normal VFS, where the file may or may not exist)

This resolver handles two paths:

- `/proc/cpuinfo` — returns static CPU info (mimicking a single-core
  virtual CPU)
- `/proc/meminfo` — returns computed memory info (512 MB total, 256 MB free)

Any other path under `/proc` (e.g., `/proc/uptime`) returns `null`, which
means the VFS will check its real filesystem — and since we didn't create
those files, the read would fail with an error.

Under the hood, resolvers are stored in a list sorted by prefix length
(longest prefix wins). When `readFile(path)` is called, the VFS:

1. Finds all resolvers whose prefix matches the path
2. Calls the most specific one (longest matching prefix) first
3. If it returns a string, use it
4. If it returns `null`, try the next resolver (if any) or fall back to
   the real filesystem

```typescript
console.log(vfs.readFile("/proc/cpuinfo").split("\n").slice(0, 4).join("\n"));
```

Reading `/proc/cpuinfo` triggers the resolver, which returns the synthetic
content. The output is sliced to show only the first 4 lines.

### Before-read hook (audit log)

```typescript
vfs.onBeforeRead("/etc", () => {
  console.log("  beforeRead: /etc file accessed");
});

vfs.writeFile("/etc/hostname", "my-vm");
console.log(vfs.readFile("/etc/hostname"));
```

`onBeforeRead(prefix, callback)` registers a hook that fires **before**
every `readFile()` call on a path under the given prefix. The callback
receives the file path as an argument.

When `/etc/hostname` is read:

1. The `beforeRead` hook fires, logging `beforeRead: /etc file accessed`
2. The file is read from the real filesystem
3. The content (`my-vm`) is returned

Hooks do not return a value — they are fire-and-forget notifications.
They cannot block the read (unlike `beforeWrite` hooks which can throw).

### Before-write hook (read-only filesystem guard)

```typescript
vfs.onBeforeWrite("/sys", (path) => {
  throw new Error(`Read-only: cannot write to ${path}`);
});

try {
  vfs.writeFile("/sys/test", "should fail");
} catch (err: unknown) {
  const msg = err instanceof Error ? err.message : String(err);
  console.log(`  Blocked: ${msg}`);
}

vfs.writeFile("/tmp/test", "allowed");
console.log(`  /tmp/test content: ${vfs.readFile("/tmp/test")}`);
```

`onBeforeWrite(prefix, callback)` registers a hook that fires **before**
every `writeFile()` under the prefix. Unlike `beforeRead`, the `beforeWrite`
hook can **throw** to abort the write — implementing a read-only guard.

- Writing to `/sys/test` triggers the hook which throws, preventing the
  write. The error is caught in the `try/catch` and printed.
- Writing to `/tmp/test` does NOT match the `/sys` prefix, so it proceeds
  normally. The content is written to the VFS and then read back.

This pattern is used to enforce immutability on specific subtrees (e.g.,
`/sys`, `/proc`) while allowing writes everywhere else.

### Hook removal

```typescript
vfs.offBeforeRead("/etc");
vfs.offBeforeWrite("/sys");
console.log("  Hooks removed");
```

`offBeforeRead(prefix)` and `offBeforeWrite(prefix)` remove the previously
registered hooks. After removal:

- Reads to `/etc` no longer trigger the audit log
- Writes to `/sys` no longer throw — they succeed against the real VFS

The prefix must match exactly the one used during registration. This allows
fine-grained removal without affecting other hooks on different prefixes.

### Multiple resolvers with priority

```typescript
vfs.registerContentResolver("/", (path) => {
  if (path === "/etc/os-release") {
    return 'NAME="Virtual Container"\nVERSION="1.0"\n';
  }
  return null;
});

console.log(vfs.readFile("/etc/os-release"));
```

A second resolver is registered with prefix `/` — the root. This resolver
handles `/etc/os-release`, a file that exists on real Linux systems but
wasn't created in our VFS.

The resolver chain now has two entries:

| Resolver | Prefix | Handles |
|----------|--------|---------|
| First | `/proc` | `/proc/cpuinfo`, `/proc/meminfo` |
| Second | `/` | `/etc/os-release` |

When reading `/etc/os-release`:

1. The VFS checks resolvers — the longest prefix match `/` wins
2. The root resolver matches and returns the content
3. No file was ever written to disk

The `/etc/os-release` resolver overlaps with the `/etc` prefix that has a
`beforeRead` hook. But resolvers and hooks are independent systems —
the hook fires on all reads under `/etc`, while the resolver only activates
when the file doesn't exist in the real VFS. If `/etc/os-release` had been
written to the real VFS, the resolver would not be consulted (real files
take priority).

### Stats of resolved files

```typescript
const cpuStat = vfs.stat("/proc/cpuinfo");
console.log(`  /proc/cpuinfo: type=${cpuStat.type}, size=${cpuStat.type === "file" ? cpuStat.size : "N/A"}`);
```

`stat()` on a dynamically resolved file returns a `FileStat` object:

- `type` is `"file"` (the resolver's return value is treated as a regular
  file, not a directory or symlink)
- `size` reflects the byte length of the resolver's return value
- `createdAt` / `modifiedAt` are set to the current timestamp (since the
  content is generated on read, not stored)

This means tools that check `stat()` before reading (like `ls -l`) will
see a valid file with a real size, even though no file exists on the VFS.

## Module Interactions

`VirtualFileSystem` is a self-contained module with no external dependencies. Content resolvers and lifecycle hooks are registered directly on the VFS instance. The resolver chain and hook system operate independently: resolvers generate dynamic content, hooks observe or guard access. Both are optional layers on top of the real filesystem store.

## Key Concepts

- **Resolver chain with fallthrough:** Resolvers are tried in order of
  specificity (longest matching prefix first). Returning `null` defers
  to the next resolver or the real filesystem. This enables layered
  virtual filesystems (like overlayfs or unionfs).
- **Hooks are fire-and-forget (read) vs throw-to-block (write):**
  `beforeRead` hooks are purely observational. `beforeWrite` hooks can
  abort via exceptions. This asymmetry matches the real use case: auditing
  reads is informative, but blocking writes requires enforcement.
- **Resolution vs hooks are independent:** A resolver generates content,
  a hook observes or guards access. You can have both on the same path
  prefix, and they operate in order: hooks fire first, then resolver is
  consulted if the file doesn't exist in the real VFS.
- **Stat works on resolved files:** The VFS invokes the resolver during
  `stat()` to determine the content length. This means resolved files
  appear as real files in directory listings and size checks.
- **Same-prefix matching:** Both resolvers and hooks use prefix matching
  on paths, not exact matches. `/proc` matches `/proc/cpuinfo`, `/proc/meminfo`,
  and any other file under that subtree.
